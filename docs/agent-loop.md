# Agent Control Loop

ArtAgents runs are file-based. Every artifact is a JSON or media file on disk,
and every action is a CLI command. This makes the toolkit controllable by any
AI agent that can read JSON and execute shell commands.

## Run Manifest

Every pipeline run creates `run.json` in the run directory. The manifest
tracks inputs, step history, artifact paths, errors, and available next
actions. It is the single source of truth for run state.

```bash
python3 -m artagents inspect-run runs/example
```

The manifest is updated automatically by the hype orchestrator at every
lifecycle point: step start, step completion, artifact creation, and
finalization.

## Protocol

An agent working with a completed run follows this loop:

### 1. Inspect

```bash
python3 -m artagents inspect-run RUN_DIR
```

Returns a single JSON payload containing:

| Field | Contents |
| --- | --- |
| `manifest` | Run ID, status, current stage, inputs, errors |
| `paths` | Relative paths to pool, arrangement, timeline, assets, render |
| `artifacts` | Every manifest-tracked artifact with existence check |
| `review` | Editor verdict, ship confidence, note count |
| `clips` | Every clip with audio/visual pool labels, text overlay, editor notes |
| `stale_after_revision` | Downstream files that would be rebuilt after edits |
| `available_actions` | What the agent can do next, with exact commands and schemas |

One invocation gives the agent a complete model of the run without needing to
locate or parse individual files.

### 2. Decide

Read the `review` section. If `verdict` is `"ship"`, the run is done. If
`verdict` is `"iterate"`, read `clips[].editor_notes` to understand what
needs fixing. Each note includes `action`, `priority`, `observation`, and
`brief_impact`.

### 3. Edit

Write a JSON array of structured edit actions and apply them:

```bash
python3 -m artagents revise RUN_DIR --edits edits.json
```

This:
- Validates every edit against the arrangement and pool
- Backs up the current arrangement to `arrangement.before-edits.json`
- Applies edits to `arrangement.json`
- Moves downstream derived outputs (timeline, assets, metadata, render,
  review) into a `drafts/` folder so they are cleanly rebuilt from the
  edited arrangement ŌĆö the previous versions are preserved as fallback
- Records the revision in `run.json`
- Sets manifest status to `needs_render`

The derived outputs are regenerated by `rerender` in the next step. The
arrangement itself ŌĆö and its backup ŌĆö are always preserved. Source files
(pool, brief, scenes, transcript, media) are never touched.

The edit vocabulary:

| Action | Required fields | What it does |
| --- | --- | --- |
| `trim` | `clip_uuid`, at least one of `start_delta_sec`/`end_delta_sec` | Shift audio trim boundaries by delta seconds |
| `replace_text` | `clip_uuid`, `text` | Change a clip's text overlay content |
| `change_style` | `clip_uuid`, `style_preset` | Change the style preset on a clip |
| `reorder` | `clip_uuid`, `order` | Move a clip to a different position |
| `delete` | `clip_uuid` | Remove a clip entirely |
| `swap` | `clip_uuid`, at least one of `audio_pool_id`/`visual_pool_id` | Replace a clip's audio or visual source |
| `insert` | `clip` (full clip object) | Add a new clip to the arrangement |

Every edit can include an optional `reason` field.

Example `edits.json`:

```json
[
  {"action": "trim", "clip_uuid": "00000001", "end_delta_sec": -0.4, "reason": "Tighten trailing pause."},
  {"action": "replace_text", "clip_uuid": "00000003", "text": "Sharper hook", "reason": "Opening title was too generic."},
  {"action": "delete", "clip_uuid": "00000007", "reason": "Redundant beat."}
]
```

### 4. Rebuild

```bash
python3 -m artagents rerender RUN_DIR
```

Reconstructs the full pipeline command from `run.json` inputs and re-runs
from `cut` onward (cut, refine, render). No need to remember the original
arguments.

Options:
- `--from STEP` ŌĆö start from a different pipeline step (default: `cut`)
- `--no-render` ŌĆö rebuild timeline and assets without the final Remotion render
- `--dry-run` ŌĆö print the reconstructed command without running it

### 5. Verify

Call `inspect-run` again to check the new state. Or generate a static HTML
review page for human oversight:

```bash
python3 -m artagents review RUN_DIR
```

The review page shows the rendered video, every clip with pool labels and
editor notes, step history, artifact previews, available actions with schemas,
and revision history.

### 6. Repeat

Loop back to step 1 until `review.verdict` is `"ship"`.

## Executor Interface

All agent capabilities are registered as standard ArtAgents executors and
appear in `python3 -m artagents executors list`:

| Executor ID | Purpose |
| --- | --- |
| `builtin.agent_context` | Returns the inspect-run JSON payload |
| `builtin.agent_actions` | Writes available actions into `run.json` |
| `builtin.apply_edits` | Applies structured edits to the arrangement |
| `builtin.rerender` | Resumes the pipeline from cut/render |

Each action returned by `inspect-run` includes the exact `command` array to
execute it and, for `apply_edits`, the full JSON Schema describing valid
input.

## What Edits Preserve

Edits only modify `arrangement.json`. Downstream generated files are moved to
`briefs/<slug>/drafts/` so they can be cleanly rebuilt ŌĆö the previous working
versions stay accessible. Edits never modify:

- `pool.json` (source material index)
- The brief file
- `scenes.json`, `transcript.json` (upstream analysis)
- Source media files
- `run.json` (updated, never deleted)

The backup at `arrangement.before-edits.json` is always written unless
`--no-backup` is passed.

## Discovery

An agent that has never seen this codebase can bootstrap by running:

```bash
python3 -m artagents --help
python3 -m artagents executors list
python3 -m artagents executors inspect builtin.agent_context --json
python3 -m artagents inspect-run RUN_DIR
```

The `available_actions` array in the inspect payload contains everything
needed to take the next step: command, executor ID, alias command, input
schema, and description.
